# 支持问题 Agent 汇报说明

## 一句话介绍

`支持问题 Agent` 是一个面向部门内部支持场景的智能协同助手。

它把 **问题登记、AI 初步分析、知识检索、结果回写、人工分派、消息提醒、汇总沉淀** 串成了一条完整流程，让原来分散、重复、依赖个人经验的支持工作，变成一个可追踪、可复用、可沉淀的标准化流程。

---

## 现在已经可以实现什么

### 1. 统一收口支持问题

- 支持大家通过飞书多维表格统一登记问题
- 问题不再散落在群聊、私聊、口头沟通和个人记录里
- 所有问题都有统一入口、统一状态、统一留痕

### 2. AI 自动完成第一轮分析

- 自动读取 `待分析` 的支持问题
- 基于现有知识库进行检索和总结
- 自动生成第一版 `AI解决方案`
- 自动补充相关文档链接
- 自动把结果回写到飞书表格

### 3. 自动识别需要人工介入的问题

- 对命中不足、置信度不够的问题，自动转为 `待人工确认`
- 支持按模块自动匹配对应支持人员
- 如果没有命中具体规则，会自动分配给兜底负责人
- 实现“AI先处理，疑难问题再交给人”

### 4. 自动通知相关人员

- 问题进入 `待人工确认` 时，自动给对应支持人员发送工作通知
- 问题进入 `人工确认完成` 时，自动给登记人发送工作通知
- 通知中可直接带飞书表格地址，支持 Web 端打开

### 5. 自动形成管理视角的汇总

- 支持立即汇总和周期汇总
- 自动发送结构化汇总邮件
- 可统计处理量、AI完成量、待人工确认量、失败量、高频问题、高频无命中主题等
- 帮助管理者看到真实问题分布和团队负载情况

### 6. 自动沉淀经验和案例

- 人工反馈、人工最终方案会同步留痕
- 后续可将“修改后采纳”“直接采纳”的问题逐步沉淀为案例
- 让团队经验从“在人脑里”转为“在系统里”

---

## 这件事为什么值得做

这个 Agent 的意义，不只是“做了一个 AI 功能”，而是把原来很多零散、重复、低附加值的支持动作自动化了。

从业务角度看，它解决的是三个核心问题：

### 1. 支持问题很多，但处理方式过于依赖人工

过去常见情况是：

- 人工反复查资料
- 人工整理答案
- 人工更新台账
- 人工提醒相关同事
- 人工再做汇总

这些工作本身重要，但大量时间消耗在重复动作上，真正用于分析问题、解决复杂问题、优化系统本身的时间反而被挤占。

### 2. 知识已经存在，但很难稳定复用

部门内往往并不是真的“没有答案”，而是：

- 文档分散
- 经验分散
- 新同学不熟
- 老同学容易被频繁打断

这个 Agent 的价值是把“已有知识”转化成“可自动调用的能力”，让知识真正服务于日常工作。

### 3. 流程缺少标准化，容易形成隐性成本

如果支持流程没有统一入口和状态管理，会带来很多管理成本：

- 问题是否被处理，不透明
- 谁负责，不清晰
- 是否重复提问，不好判断
- 哪类问题最耗时，难以统计
- 哪些知识缺口最大，不容易沉淀

现在这套 Agent，本质上是在搭建一个支持问题的“标准作业流”。

---

## 对部门提效的价值

从提效角度看，可以重点概括为以下几点：

### 1. 让研发和核心骨干更专注于真正重要的事情

- 大量重复性支持问题，不再需要每次都从零查资料、从零组织答案
- 核心研发同学可以减少被零碎问题频繁打断
- 把更多时间投入到开发、架构优化、复杂问题攻坚上

这件事的本质，不是“少回几个问题”，而是 **把高价值人力从低频打断中释放出来**。

### 2. 实现“专人做专事”

- 常规问题由 AI 先给出初步答案
- 需要人工介入的问题，自动路由给对应模块负责人
- 登记人和支持人之间形成明确的流转关系

这样可以避免：

- 谁看到谁处理
- 临时拉人帮看
- 问题在群里反复转发

让支持流程从“临时应对”变成“有序协同”。

### 3. 为团队减负

- 减少重复查文档
- 减少重复整理回复
- 减少人工催办和通知
- 减少人工做汇总统计
- 减少因信息不对称带来的返工

很多时候，团队最累的不是解决复杂问题，而是被大量琐碎问题拖住节奏。

这个 Agent 的价值，就是持续帮团队把这些琐事接住。

### 4. 提升响应速度和服务体验

- 问题登记后可以更快进入处理流程
- AI 能先给出一版标准化回答
- 该转人工时自动转人工、自动提醒
- 登记人能在问题处理完成后及时收到通知

这会带来更直接的感受：

- 响应更快
- 流程更顺
- 反馈更及时
- 感知更专业

### 5. 把个人经验变成团队能力

- 过去很多支持能力依赖“谁最熟、谁有经验”
- 现在逐步把经验沉淀到知识库、案例库、处理流程里
- 新同学也能借助系统快速进入状态

这意味着：

- 降低对少数关键人的依赖
- 降低人员变动带来的风险
- 提升团队整体的稳定性和可复制性

### 6. 为管理提供数据抓手

通过汇总和留痕，管理层可以看到：

- 当前支持问题总量
- 哪些模块问题最多
- 哪些问题 AI 能解决
- 哪些问题必须人工兜底
- 哪些主题是知识库短板
- 哪些同学/模块负载更高

这类信息非常重要，因为它不只是“看结果”，而是能反过来指导：

- 资源分配
- 知识补齐
- 培训方向
- 产品优化优先级

## 适合向领导强调的亮点

如果要在汇报时更有吸引力，可以重点强调下面几点：

### 1. 这不是单点功能，而是一条完整闭环

它不是只会“问答”，而是已经具备：

- 问题收集
- AI分析
- 人工分派
- 人工确认
- 结果通知
- 邮件汇总
- 数据沉淀

这说明它已经接近一个可实际落地的小型业务流程系统。

### 2. 它直接作用于真实工作，而不是停留在演示层

它不是单纯聊天，也不是只做一个页面展示，而是直接嵌入现有团队的真实支持流程。

换句话说，这类 Agent 做出来以后，能真正替团队接活。

### 3. 它具备持续成长空间

当前已经能解决“支持问题流转和初步处理”。

后续还可以继续往下演进，例如：

- 识别重复问题并自动归并
- 形成更完整的案例库
- 对接更多通知渠道
- 对不同模块做更精细的分派策略
- 沉淀支持知识图谱
- 做问题趋势分析和风险预警

这意味着它不是一次性投入，而是可以逐步演进成部门级的智能支持平台能力。

### 4. 这类能力具备复制价值

支持问题 Agent 跑通后，方法论是可以复用的：

- 用在测试反馈流转
- 用在实施问题处理
- 用在内部运维咨询
- 用在产品答疑
- 用在项目交付过程中的知识问答与升级分派

也就是说，这不是只服务一个场景，而是具备横向复制潜力。

---

## 可以给领导的总结表达

可以直接用下面这段作为汇报结论：

> 我们现在做的这个支持问题 Agent，不只是一个 AI 问答工具，而是在尝试把部门里大量重复、零散、依赖个人经验的支持工作，沉淀成一条可自动流转、可持续优化的标准流程。  
> 它已经能够完成问题收集、AI 初步分析、人工分派、处理结果通知和汇总沉淀，核心价值是为团队减负、提升响应效率、减少对关键人员的打扰，并把个人经验逐步转化为团队能力。  
> 从长期看，这套能力不仅能服务当前支持场景，还可以复制到更多内部协同和服务场景，具备持续放大的价值。

---

## 一句话价值总结

`支持问题 Agent` 的核心价值可以概括为：

**让重复性支持工作被系统接住，让关键人员回到更高价值的工作上，让团队经验真正沉淀下来。**
