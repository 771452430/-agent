# 支持问题 Agent 多页签处理改造记录

## 时间
- 2026-04-04 10:30（Asia/Shanghai）

## 背景与目标
- 当前支持问题 Agent 运行时只会处理 Agent 配置地址里默认指向的单个飞书分页。
- 目标是让同一个多维表格 app 下的 `开通`、`计量`、`多语`、`产品形态升级` 等页签都能被同一套支持问题 Agent 一起读取、处理、回写和同步。

## 本次完成事项
- 后端新增飞书多维表格 `table` 列表读取能力。
- 支持问题 `run graph` 改为按整个 app 聚合读取全部页签记录，而不是只读 `agent.feishu_table_id`。
- 支持问题 `feedback graph` 同步改为全页签扫描，保证人工确认、反馈事实、案例候选都覆盖全部页签。
- 单行处理链路新增来源表上下文，回写时会写回原始页签，不再默认写回 Agent 初始选中的页签。
- 给跨页签记录引入 scoped `record_id` 规则：`table_id::record_id`，避免不同页签记录在通知日志、反馈事实、案例候选里互相覆盖。
- 前端运行记录区域补充了“来源页签 / 原始 record_id”展示，方便查看是哪一个分页的数据被处理。

## 本次新增能力
- 同一个 Support Agent 现在可以一次性处理同一多维表格 app 下的多个页签。
- 人工确认通知和登记人通知会带上对应页签链接，点击后能直接打开正确分页。
- 反馈同步、digest 统计、insights 统计现在都按整份多维表格汇总，不再只统计单页签。

## 本次修复与调整
- 修复了原来只读取 `feishu_table_id` 导致其他页签完全不参与轮巡的问题。
- 修复了多页签场景下潜在的 `record_id` 撞车风险。
- 保留现有字段映射、通知、案例审核、digest 逻辑，不改变原有业务语义。

## 验证结果
- 后端 `py_compile` 通过
- 后端 `unittest backend.tests.test_support_issue_graphs` 10 项通过
- 新增“多页签一起处理”的回归测试通过
- 前端 `tsc --noEmit` 通过
- 前端 `next build` 通过
- 本地前后端已按新代码重启

## 当前结论
- 支持问题 Agent 已从“单页签处理”升级为“多页签统一处理”。
- 你现在点“立即执行”时，链路会覆盖当前多维表格 app 下的全部页签数据。

## 后续优化项
- 在 Agent 配置页补一个“处理范围”显式开关，支持“仅当前页签 / 全部页签”二选一。
- 在前端结果页增加按页签筛选和分组展示，方便排查单个分页的问题量。
- 如果后面页签字段继续分化，可以再升级成“按页签独立解析字段映射”模式。
